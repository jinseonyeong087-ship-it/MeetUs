## 1. 문서 개요

### 1.1 문서명
`AI 회의 요약 & To-Do 아카이브 시스템 - 최종 운영 명세서 v1.0`

### 1.2 문서 목적
본 문서는 AI 회의 요약 및 To-Do 아카이브 시스템의 최종 운영 기준을 정의한다. 본 문서는 발표용 자료가 아니라 서비스 설계, 구현, 운영, 테스트, 배포 시 동일하게 참조하는 기준 문서다.

### 1.3 적용 범위

| 항목 | 범위 |
|---|---|
| 서비스 범위 | 로그인 기반 회의 파일 업로드, 전사, 요약, 결정사항 추출, 개인별 To-Do 저장/조회 |
| 사용자 범위 | 가입한 개인 사용자, 워크스페이스 초대 사용자 |
| 운영 범위 | 단일 AWS 계정 내 ECS(Fargate) 기반 운영 |
| 제외 범위 | 실시간 녹음, EKS, HPA, Multi-AZ, dev/prod 분리 |

### 1.4 문서 해석 원칙

| 항목 | 기준 |
|---|---|
| 우선순위 | 본 문서의 확정 조건이 기존 초안 문서보다 우선한다 |
| 표현 원칙 | 모호한 표현을 허용하지 않는다 |
| 변경 원칙 | 본 문서의 변경은 버전 업데이트와 함께 관리한다 |

## 2. 시스템 목적 및 범위

### 2.1 시스템 목적
본 시스템은 회의 음성 파일을 업로드 받아 전체 transcript를 저장하고, 회의 요약, 결정사항, 개인별 To-Do를 생성 및 아카이브하는 협업 시스템이다.

### 2.2 핵심 제공 기능

| 기능 | 설명 |
|---|---|
| 사용자 인증 | 로그인 후 서비스 이용 가능 |
| 워크스페이스 협업 | 초대 기반으로 워크스페이스 멤버 협업 |
| 회의 등록 | 회의 메타데이터 생성 및 파일 업로드 |
| 전사 처리 | 업로드된 m4a 파일을 transcript로 변환 |
| 요약 생성 | 전체 요약 및 결정사항 생성 |
| To-Do 생성 | 개인별 액션 아이템 구조화 및 저장 |
| 결과 조회 | 회의 결과, transcript, To-Do 목록 조회 |
| To-Do 운영 | 개별 조회, 상태 변경, 필터링 |

## 5. 회의 업로드 정책

### 5.1 업로드 제약

| 항목 | 확정 기준 |
|---|---|
| 실시간 녹음 | 미지원 |
| 업로드 방식 | 파일 업로드만 허용 |
| 최대 파일 용량 | 30MB |
| 지원 포맷 | `m4a`만 허용 |
| 저장 위치 | S3 |
| 업로드 방식 | Presigned URL 기반 직접 업로드 |

### 5.2 업로드 처리 흐름
1. 사용자는 로그인 후 워크스페이스 내에서 회의를 생성한다.
2. 서버는 회의 식별자와 업로드 대상 S3 경로를 생성한다.
3. 서버는 Presigned URL을 발급한다.
4. 클라이언트는 해당 URL로 `m4a` 파일을 직접 업로드한다.
5. 업로드 성공 확인 후 회의 상태를 `UPLOADED`로 변경한다.
6. 이후 비동기 처리 파이프라인을 시작한다.

### 5.3 업로드 검증 규칙

| 검증 항목 | 기준 |
|---|---|
| 로그인 여부 | 미로그인 사용자는 업로드 불가 |
| 워크스페이스 권한 | 해당 워크스페이스 멤버만 업로드 가능 |
| 파일 확장자 | `m4a` 외 업로드 거부 |
| 파일 크기 | 30MB 초과 시 업로드 거부 |
| 저장 경로 | 워크스페이스/회의 단위로 분리 |

## 7. To-Do 데이터 운영 기준

### 7.1 AI 반환 형식
AI는 반드시 아래 JSON 배열 구조로 To-Do를 반환해야 한다.

```json
[
  {
    "assignee": "홍길동",
    "task": "보고서 작성",
    "due_date": null
  }
]
```

### 7.2 필드 규칙

| 필드 | 타입 | 필수 여부 | 기준 |
|---|---|---|---|
| `assignee` | string | 필수 | Workspace Member와 매핑 가능한 이름 |
| `task` | string | 필수 | 수행해야 할 액션 내용 |
| `due_date` | string or null | 선택 | 없으면 `null` 허용 |

### 7.3 저장 및 운영 규칙

| 항목 | 기준 |
|---|---|
| 저장 방식 | DB 정규화 테이블 저장 |
| 담당자 매핑 | Workspace Member 기준 자동 매핑 |
| 개별 조회 | 가능해야 함 |
| 상태 변경 | 가능해야 함 |
| 필터 | 담당자, 상태, 회의 기준 필터 가능해야 함 |

### 7.4 To-Do 상태 운영 기준
본 문서에서 회의 상태 ENUM은 별도로 정의하며, To-Do는 업무 추적을 위해 별도의 상태값을 가질 수 있다. 단, To-Do 상태 설계는 회의 상태 ENUM과 분리한다.

## 8. 상태 관리 기준

### 8.1 회의 상태 ENUM

```text
CREATED
UPLOADED
PROCESSING
COMPLETED
FAILED
```

### 8.2 상태 정의

| 상태값 | 정의 |
|---|---|
| `CREATED` | 회의 메타데이터만 생성된 상태 |
| `UPLOADED` | 원본 음성 파일 업로드가 완료된 상태 |
| `PROCESSING` | 전사, 요약, To-Do 추출 중인 상태 |
| `COMPLETED` | transcript, 요약, 결정사항, To-Do 저장이 모두 완료된 상태 |
| `FAILED` | 자동 재시도 한도를 초과했거나 처리 불가로 종료된 상태 |

### 8.3 상태 전이 규칙

| 현재 상태 | 다음 상태 | 조건 |
|---|---|---|
| `CREATED` | `UPLOADED` | 파일 업로드 성공 |
| `UPLOADED` | `PROCESSING` | 처리 작업 시작 |
| `PROCESSING` | `COMPLETED` | 모든 필수 산출물 저장 성공 |
| `PROCESSING` | `FAILED` | 재시도 포함 최종 실패 |

### 8.4 재시도 규칙

| 항목 | 기준 |
|---|---|
| Retry 관리 방식 | 별도 로그로 관리 |
| 자동 재시도 횟수 | 최대 3회 |
| 실패 확정 기준 | 3회 초과 시 `FAILED` |
| 상태 의미 | 상태값은 현재 위치만 표현 |
