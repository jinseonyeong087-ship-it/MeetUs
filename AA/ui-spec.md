# 화면설계서 (AA)

## 1. 목적
AI 회의 요약 및 개인별 To-Do 자동 추출 결과를 사용자에게 직관적으로 제공한다.

---

## 2. 화면 구성

### 2.1 화면 목록 정의
| 화면명 | 경로/진입 방식 | 역할 |
|---|---|---|
| 회의 업로드 화면 | 워크스페이스 메인 또는 Archive에서 `회의 생성` 선택 | 회의 메타데이터 생성, `m4a` 파일 업로드, 처리 시작 진입점 |
| 회의 목록(Archive) 화면 | 워크스페이스 진입 후 기본 목록 화면 | 회의 검색, 필터, 정렬, 상태 확인, 상세 화면 이동 |
| 회의 상세 화면 | 목록 또는 업로드 완료 후 진입 | 요약, 결정사항, transcript, 개인별 To-Do 확인 |

---

### 2.2 회의 업로드 화면

#### 주요 UI 컴포넌트
- 회의 제목 입력 필드
- 회의 일시 입력 필드
- 파일 드롭 영역
- 파일 선택 버튼
- 업로드 시작 버튼
- 업로드 진행률 바
- 상태 메시지 영역
- 업로드 정책 안내 영역 (`m4a`, 30MB 이하)
- 오류 메시지 영역

#### 사용자 인터랙션
- 사용자는 회의 제목과 일시를 입력한다.
- 사용자는 드래그 앤 드롭 또는 파일 선택으로 오디오 파일을 지정한다.
- 사용자는 `업로드 시작` 버튼으로 회의 생성과 업로드를 진행한다.
- 업로드 중 진행률을 확인한다.
- 업로드 완료 후 상세 화면으로 이동한다.
- 오류 발생 시 재업로드 또는 완료 처리 재시도를 수행한다.

#### 상태 변화
- 초기 상태: 입력 대기, 업로드 버튼 비활성 또는 필수값 충족 시 활성
- 회의 생성 완료: `CREATED`
- 파일 업로드 완료: `UPLOADED`
- 처리 시작 반영: `PROCESSING`
- 업로드 또는 처리 실패: `FAILED`

#### 화면 전환 흐름
- 워크스페이스 메인/Archive → 회의 업로드 화면
- 회의 업로드 화면 → 업로드 진행 상태 유지
- 업로드 성공 후 → 회의 상세 화면
- 오류 발생 시 → 현재 화면 유지 후 재시도 유도

#### 사용자 인터랙션 흐름
1. 제목과 일시를 입력한다.
2. `m4a` 파일을 선택한다.
3. `업로드 시작`을 누른다.
4. 회의 생성 API 호출 후 Presigned URL을 발급받는다.
5. 브라우저에서 S3로 직접 업로드한다.
6. 완료 처리 후 상세 화면으로 이동한다.

#### API 연동 정의
- `POST /meetings`
- `POST /meetings/{meetingId}/audio/presigned-url`
- `POST /meetings/{meetingId}/audio:complete`
- `POST /meetings/{meetingId}/processing:run`
- `GET /meetings/{meetingId}`

### 2.3 회의 목록(Archive) 화면

#### 주요 UI 컴포넌트
- 검색 입력창
- 상태 필터
- 기간 필터
- 정렬 드롭다운
- 새로고침 버튼
- 회의 목록 테이블 또는 카드 리스트
- 상태 뱃지
- 페이지네이션
- Empty State 영역
- 목록 조회 오류 영역

#### 사용자 인터랙션
- 사용자는 제목 기준 검색을 수행한다.
- 사용자는 상태, 기간 필터를 적용한다.
- 사용자는 최신순/오래된순 정렬을 선택한다.
- 사용자는 회의를 선택해 상세 화면으로 이동한다.
- 사용자는 새로고침으로 최신 상태를 다시 확인한다.

#### 상태 변화
- 목록 로딩 중: 스켈레톤 또는 로딩 인디케이터 표시
- 목록 조회 성공: 데이터 렌더링
- 데이터 없음: Empty State 표시
- 일부 회의 `PROCESSING`: 상태 뱃지와 진행 안내 유지
- 목록 조회 실패: 오류 메시지와 재조회 버튼 표시

#### 화면 전환 흐름
- 워크스페이스 진입 → Archive 화면
- Archive 화면 → 회의 상세 화면
- Archive 화면 → 회의 업로드 화면
- 회의 상세 화면 → Archive 화면 복귀

#### 사용자 인터랙션 흐름
1. 목록 진입 시 회의 목록을 조회한다.
2. 검색어, 필터, 정렬 조건을 적용한다.
3. 상태별 회의를 확인한다.
4. 특정 회의를 선택하여 상세 화면으로 이동한다.
5. 처리 중 회의는 새로고침 또는 polling 반영 결과를 확인한다.

#### API 연동 정의
- `GET /meetings?query=&status=&from=&to=&sort=&page=&size=`
- `GET /meetings/{meetingId}`

### 2.4 회의 상세 화면

#### 주요 UI 컴포넌트
- 회의 기본 정보 헤더
- 상태 뱃지
- 상태 안내 메시지
- 전체 요약 섹션
- 결정사항 섹션
- transcript 섹션
- 개인별 To-Do 카드/리스트 섹션
- 새로고침 버튼
- 재처리 버튼
- 스켈레톤 로딩 영역
- 오류 표시 영역

#### 사용자 인터랙션
- 사용자는 회의 상태를 확인한다.
- 사용자는 요약, 결정사항, transcript, 개인별 To-Do를 읽는다.
- 처리 중이면 자동 갱신 또는 수동 새로고침으로 상태를 확인한다.
- 실패 시 재처리를 시도한다.
- Archive로 복귀한다.

#### 상태 변화
- 진입 직후: 상세 데이터 로딩
- `CREATED`: 업로드 대기 또는 업로드 필요 안내
- `UPLOADED`: 처리 대기 안내
- `PROCESSING`: 결과 생성 중 로딩 UI 유지
- `COMPLETED`: 결과 데이터 전체 노출
- `FAILED`: 실패 메시지와 재처리 액션 노출

#### 화면 전환 흐름
- 업로드 화면 → 회의 상세 화면
- Archive 화면 → 회의 상세 화면
- 회의 상세 화면 → Archive 화면

#### 사용자 인터랙션 흐름
1. 상세 진입 시 회의 상태와 결과를 조회한다.
2. `PROCESSING` 상태면 자동 polling으로 상태를 갱신한다.
3. `COMPLETED` 상태 전환 시 결과 섹션을 렌더링한다.
4. `FAILED` 상태면 오류 메시지와 재처리 버튼을 노출한다.
5. 사용자는 Archive로 복귀하거나 수동 새로고침을 수행한다.

#### API 연동 정의
- `GET /meetings/{meetingId}`
- `GET /meetings/{meetingId}/result`
- `GET /meetings/{meetingId}/todos`
- `POST /meetings/{meetingId}/processing:run`

### 예외 처리 UI
- 처리 실패 안내 메시지
- 재처리 버튼
- 상세 조회 실패 시 새로고침 버튼
- 결과 없음 시 Empty State 표시

## 3. 공통 UI 규칙

### 3.1 상태 뱃지 및 공통 표시 규칙
- 상태 뱃지: `CREATED`, `UPLOADED`, `PROCESSING`, `COMPLETED`, `FAILED`
- 로딩 상태에서는 스켈레톤 또는 진행 인디케이터를 사용한다.
- API 실패 시 토스트와 인라인 오류 메시지를 함께 제공한다.
- 목록과 상세에서 동일한 상태 명칭과 색상 체계를 사용한다.
- 상태 메시지는 사용자가 현재 단계와 다음 행동을 바로 이해할 수 있도록 작성한다.

### 3.2 상태별 UI 동작
| 상태 | 표시 메시지 | 버튼 활성화 | 로딩 표시 |
|---|---|---|---|
| `CREATED` | `회의가 생성되었습니다. 오디오 파일을 업로드해주세요.` | 업로드 시작 활성 | 없음 |
| `UPLOADED` | `업로드가 완료되었습니다. AI 분석을 준비 중입니다.` | 새로고침 활성, 재처리 비활성 | 부분 로딩 가능 |
| `PROCESSING` | `전사, 요약, To-Do를 생성 중입니다.` | 새로고침 활성, 편집/재처리 비활성 | 스피너 + 스켈레톤 표시 |
| `COMPLETED` | `분석이 완료되었습니다.` | 새로고침 활성, 결과 탐색 활성 | 없음 |
| `FAILED` | `처리 중 오류가 발생했습니다. 다시 시도해주세요.` | 재처리 활성, 새로고침 활성 | 없음 |

### 3.3 상태별 화면 반영 기준
- `CREATED`: 업로드 관련 컴포넌트를 노출하고 결과 영역은 숨긴다.
- `UPLOADED`: 업로드 완료 메시지와 처리 대기 안내를 노출한다.
- `PROCESSING`: 결과 영역 대신 스켈레톤과 진행 메시지를 표시한다.
- `COMPLETED`: 결과 컴포넌트를 모두 활성화하고 상태 polling을 종료한다.
- `FAILED`: 오류 영역을 우선 노출하고 재처리 버튼을 제공한다.

### 3.4 API 연동 공통 기준
| 화면 | 주요 API | 목적 |
|---|---|---|
| 회의 업로드 화면 | `POST /meetings` | 회의 메타데이터 생성 |
| 회의 업로드 화면 | `POST /meetings/{meetingId}/audio/presigned-url` | S3 업로드 URL 발급 |
| 회의 업로드 화면 | `POST /meetings/{meetingId}/audio:complete` | 업로드 완료 처리 |
| 회의 업로드 화면 | `POST /meetings/{meetingId}/processing:run` | AI 처리 시작 |
| 회의 목록(Archive) 화면 | `GET /meetings` | 목록 조회, 검색, 필터, 정렬 |
| 회의 상세 화면 | `GET /meetings/{meetingId}` | 상세 상태 및 기본 정보 조회 |
| 회의 상세 화면 | `GET /meetings/{meetingId}/result` | transcript, 요약, 결정사항 조회 |
| 회의 상세 화면 | `GET /meetings/{meetingId}/todos` | 개인별 To-Do 조회 |

### 3.5 UX 고려 사항
- 업로드 진행 상태는 퍼센트와 단계 메시지를 함께 표시한다.
- `PROCESSING` 상태에서는 `자동으로 결과가 갱신됩니다.` 안내를 제공한다.
- 장시간 처리 시 사용자가 이탈해도 Archive에서 상태를 다시 확인할 수 있어야 한다.
- 데이터 없음 상태는 단순 빈 화면이 아니라 다음 행동을 안내하는 문구를 포함한다.
- 오류 메시지는 기술 원인보다 사용자 행동 기준으로 작성한다.
- 실패 후 재시도 경로는 명확한 버튼으로 제공한다.
- 목록과 상세 모두 수동 새로고침 경로를 제공한다.

### 3.6 Empty State 및 오류 메시지 기준
- 업로드 화면 Empty State: 파일 선택 전 기본 안내 문구 표시
- Archive Empty State: `등록된 회의가 없습니다. 새 회의를 생성해주세요.`
- 상세 Empty State: 결과 생성 전 `분석 결과가 아직 준비되지 않았습니다.`
- 네트워크 오류: `일시적인 오류가 발생했습니다. 다시 시도해주세요.`
- 권한 오류: `이 회의에 접근할 수 없습니다.`

## 4. 반응형 기준

### 4.1 공통 기준
- Desktop 우선 설계
- Tablet, Mobile에서는 단일 컬럼 중심으로 재배치
- 주요 액션 버튼은 화면 하단 또는 상단 고정 영역에 우선 배치
- 상태 뱃지와 상태 메시지는 모든 해상도에서 첫 화면에 보여야 한다

### 4.2 Desktop
- 회의 업로드 화면은 입력 영역과 정책 안내 영역을 2열로 배치한다.
- Archive 화면은 테이블 중심 레이아웃을 사용한다.
- 상세 화면은 요약/결정사항과 transcript/To-Do를 분리한 다단 레이아웃을 사용할 수 있다.

### 4.3 Tablet
- 회의 업로드 화면은 입력 영역과 진행 상태 영역을 상하 배치한다.
- Archive 화면은 테이블 축소형 또는 2열 카드 리스트로 전환한다.
- 상세 화면은 주요 정보와 결과 섹션을 세로 스택으로 재배치한다.

### 4.4 Mobile
- 회의 업로드 화면은 단일 컬럼으로 구성하고 업로드 버튼을 하단 고정형으로 배치한다.
- Archive 화면은 카드 리스트 레이아웃으로 전환한다.
- 필터와 정렬은 드로어 또는 접이식 패널로 축소한다.
- 상세 화면은 요약, 결정사항, transcript, To-Do를 세로 순서로 배치한다.
- 긴 transcript는 접기/펼치기 또는 내부 스크롤 영역을 사용한다.
